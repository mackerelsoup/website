# Architecture

A SvelteKit app that fronts a WebDAV file store ("the cloud drive") with a chunked, resumable upload
pipeline and live save-progress over SSE. This doc maps every file/component to its purpose and shows
how they connect, so you can find the right place to make a change without re-deriving the flow from
scratch each time.

## Tech stack

- **SvelteKit** (Svelte 5 runes) with `@sveltejs/adapter-node` ‚Äî deploys as a standalone Node server,
  not serverless/static.
- **better-auth** for auth (Postgres-backed, email/password + GitHub), wired into `hooks.server.ts`.
- **Drizzle ORM** + Postgres for the database.
- **WebDAV** (`webdav` npm client) as the actual file storage backend ‚Äî the app is a UI/API layer in
  front of a WebDAV server, not a filesystem of its own.
- **SSE (Server-Sent Events)** for pushing upload/save progress from server to client.

## Directory layout

```
src/
  routes/           file-based routes: pages + API endpoints
  lib/               client-safe shared code ($lib/*)
  lib/server/         server-only shared code ($lib/server/*, never bundled to the client)
  lib/assets/          static assets bundled through Vite
  hooks.server.ts      global request hook (auth session population)
  app.d.ts              ambient types (extends App.Locals with better-auth user/session)
static/                 files served as-is (robots.txt)
```

## Route tree

| Route | Files | Purpose |
|---|---|---|
| `/` | `+page.svelte` | Unused SvelteKit starter placeholder ‚Äî not linked from the cloud UI. |
| `/cloud` | `+page.server.ts`, `+page.svelte` | Health-check landing page: pings the WebDAV backend (`client.exists('/')`); redirects to `/cloud/files` if reachable, otherwise renders `components/Offline.svelte`. |
| `/cloud/files` | `+page.server.ts`, `+page.svelte`, `upload.svelte.ts` | The main file browser ‚Äî directory listing, rename/delete/bulk-delete, and the upload picker/progress banner. |
| `/cloud/files/upload/init` | `+server.ts` | `POST` ‚Äî starts or resumes a chunked upload transfer. |
| `/cloud/files/upload/chunk` | `+server.ts` | `PUT` ‚Äî receives one chunk of a file; triggers finalize when the last chunk lands. |
| `/cloud/files/upload-progress` | `+server.ts` | `GET` (SSE) ‚Äî streams server-side save progress for an upload session. |
| `/cloud/file` | `+server.ts` | `GET` ‚Äî streams a single file's raw bytes from WebDAV for inline view/download (`?path=`). |

No `/auth`, `/login`, or `/api/auth` route folder exists ‚Äî better-auth's routes are handled implicitly
inside `hooks.server.ts` via `svelteKitHandler`. **There is currently no login/signup UI, and no
route-level auth guard ‚Äî `/cloud/*` is unauthenticated in the current code.**

## `src/lib` (client-safe)

- **`upload-constants.ts`** ‚Äî `CHUNK_SIZE` (5MB), `computeTransferId()` (stable id hashed from
  `destPath::filename::size::lastModified`, via `crypto.subtle.digest` with a hand-rolled fallback for
  non-HTTPS origins), `totalChunks(size)`. Shared between client and server ‚Äî hence not under `server/`.
- **`webdav.ts`** ‚Äî the storage layer. Thin wrapper around the `webdav` npm client:
  `listDirectory`, `readFile`, `writeFile`, `deleteItem`, `moveItem`, `createDirectory`, `exists`.
  Every WebDAV read/write in the app funnels through here.
- **`tailscale.ts`** ‚Äî checks whether a named device is `connectedToControl` via the Tailscale API.
  Not currently wired into the `/cloud` online check (which talks to WebDAV directly instead).

## `src/lib/server` (server-only)

- **`auth.ts`** ‚Äî configures `better-auth`: Postgres via `drizzleAdapter`, email/password + GitHub
  login, SvelteKit cookie plugin.
- **`db/index.ts`** ‚Äî Drizzle Postgres client (`DATABASE_URL`).
- **`db/schema.ts`** ‚Äî defines a `task` table (unused demo leftover) and re-exports `auth.schema.ts`.
- **`db/auth.schema.ts`** ‚Äî placeholder; meant to be generated by `npm run auth:schema`
  (`@better-auth/cli generate`) to produce the real `user`/`session`/`account` tables. **Not generated
  yet** ‚Äî flagged as a gap below.
- **`upload-state.ts`** ‚Äî SSE session registry: in-memory `Map<uploadId, UploadSession>`, each holding
  an `EventEmitter` plus a replay buffer of events (`saving` / `saved` / `complete` / `error`).
  `getOrCreateUploadSession` exists because the SSE client connects *before* `/upload/init` runs.
  Includes a 30-minute TTL GC (`cleanupStaleSessions`).
- **`chunk-transfer.ts`** ‚Äî chunked-upload state machine. In-memory `Map<transferId, ChunkTransfer>`
  plus chunk files on disk at `os.tmpdir()/cloud-uploads/<transferId>/chunk.<n>`.
  `createChunkTransfer` (idempotent, re-creates if already `done` to support re-uploads),
  `receivedChunks` (resume support), `writeChunk`, `isComplete`, `assemble`, `claimFinalize`
  (compare-and-set so exactly one caller finalizes), `markDone`, `cleanupStaleTransfers` (30-min TTL).
- **`finalize-upload.ts`** ‚Äî `finalizeFile()`: assembles chunks, writes the file to WebDAV
  (`createDirectory` + `writeFile`), emits `saving` ‚Üí `saved` ‚Üí (if last file in the batch) `complete`
  over SSE. On failure, releases the finalize claim and emits `error` instead of throwing, so the route
  never 500s mid-upload.
- **`safe-path.ts`** ‚Äî `resolveUploadPath(destPath, filename)`: path-traversal guard so an uploaded
  file can never resolve outside `destPath` (folder uploads may nest deeper, never escape via `../`).

## The upload/save pipeline

This is the part most worth understanding end-to-end, since it spans client, two HTTP endpoints, an
SSE stream, and three pieces of server state.

```
+page.svelte
   ‚îÇ user picks files
   ‚ñº
upload.svelte.ts (UploadManager.start)
   ‚îÇ 1. opens EventSource(/cloud/files/upload-progress?id=uploadId)   ‚Üê ephemeral, SSE-only id
   ‚îÇ 2. per file: computeTransferId() ‚Üí stable id from path+name+size+mtime
   ‚ñº
POST /upload/init  ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚ñ∫ upload-state.ts (session)  +  chunk-transfer.ts (transfer)
   ‚îÇ returns already-received chunk indices (resume) or { done: true } if already finalized
   ‚ñº
PUT /upload/chunk  (repeated, one call per remaining chunk, retried up to 3x)
   ‚îÇ writeChunk() ‚Üí on last chunk, isComplete() triggers:
   ‚ñº
finalize-upload.ts (finalizeFile)
   ‚îÇ claimFinalize() ‚Üí assemble() ‚Üí webdav.writeFile()/createDirectory()
   ‚îÇ emits saving ‚Üí saved ‚Üí complete over the session's EventEmitter
   ‚ñº
upload-progress/+server.ts (SSE)  ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚ñ∫  UploadManager (client)
   ‚îÇ streams buffered + live events, closes stream on complete/error
   ‚ñº
UploadManager updates phase/savingIndex/savingFilename for the progress banner,
then calls invalidateAll() to refresh the directory listing.
```

Key design points to know before touching this code:

- **Two different IDs**: `uploadId` (per upload *batch*, SSE channel only) vs `transferId` (per *file*,
  derived deterministically so re-uploading the same file/path/size/mtime resumes instead of
  duplicating).
- **Client-side event queueing**: `UploadManager` buffers incoming SSE events and only replays them
  once every chunk of every file has been sent (`allChunksSent`), so the UI shows "uploading" then
  "saving" as distinct phases instead of interleaving them.
- **EventSource close race** (see `upload.svelte.ts`): the client must call `es.close()` the moment a
  `complete`/`error` event is *received*, not after it's done being animated in the drain queue ‚Äî
  otherwise the server ends the HTTP stream first, the browser's native SSE auto-reconnect kicks in,
  and `getOrCreateUploadSession` replays the entire event history, causing the progress UI to loop.
- **Finalize is claimed, not just triggered**: both the chunk endpoint (last chunk received) and the
  init endpoint (resume where every chunk was already on disk) can independently observe "transfer
  complete." `claimFinalize()` in `chunk-transfer.ts` ensures only one of them actually finalizes.
- **Temp storage vs. final storage**: in-flight chunks live under `os.tmpdir()/cloud-uploads/`; the
  only durable destination is WebDAV via `lib/webdav.ts`. Nothing in the upload pipeline touches disk
  paths that aren't temp scratch space.

## Config files that matter architecturally

- **`svelte.config.js`** ‚Äî `adapter-node` (standalone Node server).
- **`vite.config.ts`** ‚Äî `@tailwindcss/vite`, `sveltekit()`, `vite-plugin-devtools-json`.
- **`hooks.server.ts`** ‚Äî single global `Handle`: populates `event.locals.session`/`user` via
  `auth.api.getSession()`, then delegates to `svelteKitHandler` (which transparently handles
  better-auth's own routes, e.g. `/api/auth/*`, with no dedicated route folder).
- **`drizzle.config.ts`** ‚Äî schema at `lib/server/db/schema.ts`, `postgresql` dialect, `DATABASE_URL`.
- **`package.json` scripts** ‚Äî `db:push/generate/migrate/studio` (Drizzle Kit), `auth:schema`
  (regenerates `lib/server/db/auth.schema.ts`).
- **`compose.yaml`** ‚Äî local Postgres dev DB (`db:start`).
- **Env vars** ‚Äî `WEBDAV_URL`, `WEBDAV_PASSWORD` (`$env/static/private`); `DATABASE_URL`,
  `BETTER_AUTH_SECRET`, `ORIGIN`, `GITHUB_CLIENT_ID`/`GITHUB_CLIENT_SECRET` (`$env/dynamic/private`).

## Known gaps

- **No route-level auth guard.** `better-auth` is fully configured, but nothing under `src/routes`
  reads `locals.user`/`locals.session` ‚Äî `/cloud/*` is currently open to anyone.
- **`auth.schema.ts` is unpopulated.** Run `npm run auth:schema` to generate the real `user`/`session`/
  `account` tables before auth can actually persist sessions to Postgres.
- **`db/schema.ts`'s `task` table** is leftover demo scaffolding, not used by the app.
- **No sharing feature** (`/share`, public links, etc.) exists yet.
- **"Crypto" in this codebase is not encryption** ‚Äî the `computeTransferId`/`generateUUID` fallbacks
  exist only because `crypto.subtle`/`crypto.randomUUID` are unavailable on plain-HTTP (non-secure)
  origins; there is no application-level encryption layer.
